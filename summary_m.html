各储备项目退出方向按 5 层级联动（层级 1 我方两家上市公司 → 层级 2 3 家拟收购上市公司择优 1 家（排除爱朋，含双主业跨界承接）→ 层级 3 双主业/三平台架构 → 层级 4 扩大新收上市公司范围（极少使用兜底）→ 层级 5 培育后独立上市）基于客观事实单独判断。新收上限 = 1 家，总接收架构 = 3 平台（我方 2 + 新收 1），天保基建主营锁定医药（芯片可作为第二主业承接元旭），爱朋冲突不首选，新收的上市公司和我方天保基建都可双主业跨界承接多项目，几乎不需要扩大新收范围。通用分析工具：

  • 层级 1 我方两家上市公司：天保基建（主营锁定医药，柯渡/信鸿已锁定；芯片可作为第二主业承接元旭）+ 天保能源（主营能源，宏远/联通云IDC 已锁定）
  • 层级 2 3 家拟收购上市公司择优 1 家（排除爱朋，含双主业跨界承接）：科华控股（汽车涡轮增压器零部件）+ 绿田机械（通用动力机械 + 海外渠道）—— 两方案 B/C 对比；收购后可双主业跨界承接（绿田+换电=可行性高承接迅巢 / 绿田+新材料=高承接中复 / 绿田+芯片=中等承接元旭）
  • 层级 3 双主业/三平台架构：天保基建 + 天保能源 + 新收 1 家（共 3 平台承担双主业/三主业，降低交易复杂度；天保基建可双主业承接医药+芯片）
  • 层级 4 扩大新收上市公司范围（极少使用兜底）：仅当双主业跨界承接均不可行时启动（Issue 21 后几乎不使用）

扩展机制（后续项目入库流程）

当投资部新增储备项目立项时，按以下流程纳入本汇总：

  1. 新项目立项：项目方提交立项材料
  2. 单独完成纵横分析：新建 reports/{project_name}/report_main.md
  3. 纳入本汇总：在本汇总页 8 个维度表格中添加新增项目行
  4. 更新版本号：第 2 期 → 第 3 期... 项目数从 7 → N
  5. 归档历史版本：保存至 _reserved-projects/version_history/ 子目录
  6. 重新部署：通过部署脚本推送至 GitHub Pages
